# PRD: Completar el portafolio desde la hoja de vida de Javier Espinosa

## Planteamiento del problema

El portafolio actual ya comunica una propuesta fuerte alrededor de Full-Stack, ERP, e-commerce, automatización e IA, pero todavía no representa toda la amplitud de la hoja de vida de Javier Espinosa Pico. La hoja de vida muestra una trayectoria más completa: arquitectura cloud, Django, FastAPI, Java, Spring Boot, Spring Security, JUnit, Mockito, Swagger/OpenAPI, Docker, AWS, despliegues por SSH, GitHub, QA con k6 y Playwright, analítica con Google Tag Manager y Google Analytics, proyectos de reservas hoteleras, logística, sostenibilidad y control de acceso educativo.

La brecha principal es de contenido y posicionamiento: el portafolio puede parecer más estrecho de lo que realmente es la experiencia profesional. Hay proyectos, habilidades, resultados, certificaciones y matices técnicos que no están visibles o están resumidos de forma insuficiente. Esto reduce la capacidad del sitio para convencer a reclutadores, clientes y equipos técnicos de que Javier puede liderar soluciones empresariales complejas de punta a punta.

También existen inconsistencias puntuales entre la hoja de vida y el portafolio que deben resolverse antes de seguir construyendo encima: fechas, nivel de inglés, certificaciones, tecnologías incluidas en cada experiencia y profundidad de resultados. El objetivo no es copiar la hoja de vida literalmente, sino transformar su contenido en una experiencia de portafolio clara, navegable, bilingüe, verificable y orientada a casos de negocio.

## Solución

Actualizar el portafolio para que funcione como una versión interactiva, estructurada y más persuasiva de la hoja de vida. La solución debe ampliar el contenido profesional existente, añadir casos de estudio faltantes, reforzar la matriz de habilidades, corregir datos inconsistentes y presentar evidencias por experiencia, tecnología, resultado y tipo de proyecto.

El portafolio debe permitir que una persona entienda rápidamente tres cosas:

- Qué problemas empresariales resuelve Javier: integración ERP/e-commerce, CRM, automatización con IA, sistemas web a medida, control de acceso, logística, reservas, trazabilidad y QA.
- Con qué stack los resuelve: Java, Spring Boot, Python, Django, FastAPI, PHP, Laravel, Livewire, React, Next.js, Vue, TypeScript, JavaScript, HTML5, CSS3, Tailwind CSS, Bootstrap, Odoo, SAP Business One, Magento, Docker, AWS, Linux, PostgreSQL, MySQL, SQL, Power BI, Playwright, k6, Git, GitHub y herramientas de IA.
- Qué impacto logró: integridad de datos históricos, reducción de trabajo manual, digitalización de procesos, sincronización de inventario, automatización postventa, reducción de tiempos de carga, despliegues productivos, pruebas de rendimiento y trazabilidad operativa.

La implementación debe conservar el lenguaje visual interactivo del sitio actual, pero hacer que el contenido profesional tenga más densidad y mejores rutas de lectura. La experiencia debe seguir siendo bilingüe en español e inglés.

## Decisiones de implementación

- Se ampliará el modelo de contenido profesional para cubrir con mayor precisión perfil, experiencia, habilidades, educación, certificaciones, métricas, resultados y tecnologías.
- Se corregirán inconsistencias detectadas entre la hoja de vida y el contenido actual: periodo de Industrias Bicicletas Milan hasta mayo de 2026, nivel de inglés A2 si la hoja de vida es la fuente de verdad, y certificaciones listadas.
- Se añadirá o reforzará una categoría específica de QA, testing y seguridad con Playwright, k6, JUnit, Mockito, Spring Security, JWT y Swagger/OpenAPI.
- Se añadirá o reforzará una categoría de backend empresarial con Java, Spring Boot, Lombok, Spring Data JPA, Django, FastAPI, Laravel, Livewire, Node.js y APIs RESTful.
- Se añadirá o reforzará una categoría de frontend y UI con React, Next.js, Vue.js, TypeScript, JavaScript, HTML5, CSS3, Tailwind CSS y Bootstrap.
- Se añadirá o reforzará una categoría de infraestructura y despliegue con AWS, Docker, Linux, SSH, Git, GitHub, despliegues productivos y configuración de entornos web.
- Se añadirá o reforzará una categoría de analítica y marketing técnico con Google Tag Manager, Google Analytics, SEO, Power BI, medición de rendimiento y reducción de tiempos de carga.
- Se mantendrán los casos existentes de ERP/e-commerce, marketing con IA y CRM, pero se revisarán para incorporar datos de la hoja de vida que hoy no están visibles.
- Se añadirá un caso o subcaso para el portal secundario de Isavital construido con React, Java, Spring Boot, Spring Data JPA, Spring Security, JWT, JUnit y Swagger/OpenAPI.
- Se ampliará el caso de Isavital para mostrar CRM, gestión de clientes, programación de citas, Jira, Git, frontend tipado, portal desacoplado, automatización postventa con IA, seguimiento predictivo, reducción de tiempos de respuesta e impacto en retención y satisfacción del cliente.
- Se añadirá un caso de reservas hoteleras con pasarelas de pago, RBAC, Laravel, Livewire y MySQL.
- Se añadirá un caso de plataforma logística para cadena de suministro e importación con Vue.js, JavaScript y Bootstrap.
- Se añadirá un caso de plataforma de sostenibilidad para trazabilidad, recolección y reutilización de residuos con Next.js, React, SSR y experiencia responsive.
- Se añadirá un caso de control de acceso y asistencia estudiantil para SENA con Laravel, Vue.js, Bootstrap, alta concurrencia, gestión administrativa, registro y monitoreo de asistencia, y reducción de tiempos de ingreso al centro educativo.
- Los proyectos confidenciales se presentarán como estudios de caso sin enlaces públicos, código ni información sensible de clientes.
- Cada caso de estudio tendrá al menos: contexto de negocio, reto, solución, arquitectura, stack, responsabilidades de Javier, resultados y limitaciones de confidencialidad.
- El resumen del hero deberá reflejar un posicionamiento más amplio: Full-Stack empresarial, integración ERP/e-commerce, cloud, datos, QA e IA aplicada.
- Las métricas del hero deberán evitar afirmaciones no verificadas y priorizar cifras presentes en la hoja de vida: más de 5 años de experiencia, 10 años de datos históricos, reducción del 40% de trabajo manual, digitalización del 100% de gestión de clientes y citas.
- Los resultados derivados de la hoja de vida deberán distinguir entre métricas cuantificadas y resultados cualitativos. Las métricas cuantificadas incluyen 10 años de datos históricos, reducción del 40% del trabajo manual de marketing y digitalización del 100% de gestión de clientes y citas. Los resultados cualitativos incluyen erradicación de errores de stock, reducción de tiempos de carga, reducción de tiempos de respuesta postventa, mejora de retención y satisfacción, y reducción de tiempos de ingreso en el SENA.
- La navegación actual debe seguir siendo simple, pero deberá permitir acceso rápido a experiencia, habilidades, educación, casos y contacto.
- La estructura bilingüe debe mantenerse. Todo contenido nuevo en español deberá tener equivalente en inglés antes de considerarse terminado.
- El SEO deberá actualizar títulos, descripciones, palabras clave implícitas, datos estructurados y textos sociales para reflejar el perfil completo.
- El diseño debe priorizar legibilidad y escaneo. No se deben añadir bloques extensos de texto sin jerarquía visual.
- La experiencia móvil debe tratarse como requisito principal, no como ajuste posterior, porque reclutadores suelen revisar perfiles desde dispositivos pequeños.
- La arquitectura de contenido debe seguir siendo mantenible: el contenido profesional debe vivir separado de la lógica de presentación y las secciones visuales deben consumir datos estructurados.
- Los módulos visuales existentes deben reutilizarse cuando sea posible, agregando variantes solo si la nueva información realmente lo exige.
- Se podrá añadir una matriz de capacidades que conecte tecnología, años o contexto, proyectos relacionados y nivel de exposición, siempre que no se convierta en una tabla difícil de leer en móvil.
- Se deberá revisar que el uso de "arquitecto" y "lideré" esté respaldado por responsabilidades concretas en la hoja de vida, para mantener credibilidad.
- Se deberá evitar inflar el nivel de inglés o certificaciones no listadas en la hoja de vida salvo que Javier confirme que son vigentes.

## Decisiones de testing

- Un buen test debe validar comportamiento externo visible para usuarios o contratos de contenido, no detalles internos de implementación.
- Se probará que la aplicación compile correctamente con el contenido ampliado y que no existan errores de tipos en la estructura de portafolio, proyectos, previews, traducciones y SEO.
- Se probará que todos los proyectos definidos aparezcan en listados, rutas de detalle, navegación de siguiente proyecto y filtros si existen.
- Se probará que cada proyecto tenga versión en español e inglés, preview, categoría, stack, resultados y estado de confidencialidad cuando aplique.
- Se probará que las secciones de experiencia, habilidades, educación, certificaciones y contacto rendericen con listas largas sin romper el layout.
- Se probará responsive en escritorio y móvil para hero, timeline de experiencia, matriz de habilidades, listados de proyectos y páginas de detalle.
- Se probará que los textos largos no se salgan de tarjetas, botones, etiquetas o columnas.
- Se probará que las rutas localizadas y metadatos SEO se generen con títulos y descripciones actualizadas.
- Se probará que la información de contacto siga visible y funcional después de ampliar el contenido.
- Se probará accesibilidad básica: orden de encabezados, etiquetas de navegación, contraste suficiente, foco interactivo y contenido comprensible sin depender de animaciones.
- Se probará que los proyectos confidenciales no muestren enlaces de demo o código cuando el estado indique que son privados.
- Se recomienda usar pruebas de tipo y build como verificación mínima obligatoria.
- Se recomienda usar pruebas E2E ligeras para validar navegación principal, cambio de idioma, apertura de proyectos y contacto.
- Se recomienda usar revisión visual manual o capturas con navegador para verificar que las secciones nuevas mantienen el estilo actual.

## Fuera de alcance

- No se implementarán cambios de código dentro de este PRD; este documento define el alcance para una fase posterior.
- No se publicará el portafolio ni se hará despliegue a producción.
- No se crearán commits, issues remotos ni pull requests.
- No se expondrá código privado, credenciales, datos sensibles de empresas, capturas internas ni información confidencial de clientes.
- No se inventarán métricas que no estén en la hoja de vida o que Javier no confirme.
- No se rediseñará completamente la identidad visual del sitio.
- No se reemplazará la experiencia 3D existente salvo que sea estrictamente necesario para mejorar legibilidad o rendimiento.
- No se agregará un blog, CMS o panel administrativo.
- No se cambiará el stack principal del portafolio.
- No se hará una nueva hoja de vida descargable salvo que Javier lo pida como alcance separado.
